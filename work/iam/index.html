Introducing RBAC to the DigitalOcean Platform

Built the identity and access management infrastructure that introduced role-based access control across the DigitalOcean platform, sitting in the critical path of the control plane and serving billions of authorization decisions per day with sub-millisecond response times.

Context

As DigitalOcean expanded its customer base to include larger teams and enterprises, the lack of fine-grained access control became a significant gap. Users needed a way to collaborate on infrastructure without sharing credentials or granting full account access — a foundational requirement for any serious cloud platform.

I worked as a staff engineer leading the design and implementation of IAM infrastructure that introduced RBAC (role-based access control) to the DigitalOcean control plane. This wasn't a greenfield service — it had to be woven into an existing platform serving millions of customers, without downtime and without breaking existing workflows.

Problem

DigitalOcean's existing authorization model was coarse-grained: you either had full access to an account or you didn't. As teams grew and compliance requirements tightened, this became untenable. Customers needed to grant team members access to specific resources and actions — and revoke it reliably.

The new IAM system had to integrate deeply with the existing control plane. Every API request and UI action flowing through the platform needed to pass through authorization checks. Latency was non-negotiable — adding even a few milliseconds to every request would have a compounding effect at scale.

Goals

  • Introduce role-based access control across all DigitalOcean products
  • Sub-millisecond authorization decision latency
  • Seamless integration with existing API and control plane flows
  • Backwards compatibility — no disruption to existing customers
  • Audit logging for all access decisions
  • Self-serve team and permissions management via the UI and API

Approach

The authorization engine was built as a dedicated gRPC service, deployed independently and accessed by every component in the control plane that needed to make access decisions. The design prioritized read performance — the vast majority of operations are authorization checks, not writes.

Permission data was structured to allow efficient lookups by subject (user or token), resource, and action. Aggressive caching at multiple layers — both within the authorization service and in calling services — kept the hot path sub-millisecond. Cache invalidation was driven by event streams to ensure consistency without sacrificing performance.

Backwards compatibility was handled by treating existing full-access accounts as implicitly holding a root-level role. This meant the new system could be deployed incrementally without requiring customers to migrate or reconfigure anything.

Key Challenges

Latency in the Critical Path

Problem: Authorization checks were required for virtually every API request. Any latency regression would ripple across the entire platform.

Solution: Combined in-process caching in calling services with a high-performance gRPC authorization service. Cache TTLs were tuned aggressively, and invalidation was event-driven rather than poll-based to keep data fresh without sacrificing speed.

Trade-off: Short window of potential staleness on permission changes, mitigated by event-driven invalidation with bounded lag.

Backwards Compatibility at Scale

Problem: Millions of existing customers relied on implicit full-account access. Any change to the authorization model risked breaking existing integrations.

Solution: Modeled existing access patterns as implicit roles, allowing the new RBAC system to be the single source of truth without requiring a migration. New capabilities were additive — customers opted in to fine-grained control at their own pace.

Audit Logging Without Overhead

Problem: Recording every authorization decision at billions of decisions per day risked becoming a write bottleneck.

Solution: Audit events were published asynchronously to Kafka, decoupling the decision path from the logging path entirely. Consumers downstream handled persistence and indexing.

Results & Impact

  • Billions of authorization decisions per day
  • <1ms p99 decision latency
  • RBAC adopted across all DigitalOcean products
  • Unblocked enterprise sales requiring team-based access control
  • Zero downtime migration for millions of existing customers

My Contributions

  • Led architecture and design of the IAM system
  • Built the authorization engine and gRPC service
  • Designed the permission model and caching strategy
  • Implemented event-driven cache invalidation
  • Coordinated integration across platform teams
  • Drove the backwards-compatible migration strategy

Lessons Learned

The hardest part of IAM isn't the authorization logic — it's the data model. Getting the resource hierarchy and action space right early matters enormously, because changing them later requires migrating permissions data in production. Investing in a clean, extensible model upfront paid significant dividends as new products were onboarded.
