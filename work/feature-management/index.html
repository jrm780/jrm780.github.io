Building a Low-Latency Feature Management Platform at DigitalOcean

Designed and built an in-house feature flag system enabling safe feature rollouts and A/B testing across the DigitalOcean platform, serving billions of evaluation requests per day with sub-millisecond response times.

Context

As DigitalOcean's engineering organization grew, the need for safe, incremental feature rollouts became critical. Teams needed a way to gate new functionality behind flags — controlling exposure by user, team, region, or arbitrary percentage — without deploying separate code paths or coordinating risky big-bang releases.

I led the design and implementation of an in-house feature management platform built to serve this need at DigitalOcean's scale: billions of flag evaluations per day, with the latency bar of an infrastructure system sitting in the hot path of user-facing requests.

Problem

Off-the-shelf feature flag solutions either couldn't meet the latency requirements or introduced unacceptable vendor dependencies for a system this critical. Existing internal tooling was ad-hoc, inconsistent across teams, and had no support for targeting rules, gradual rollouts, or experimentation.

The system needed to be blazing fast — flag evaluation had to add effectively zero overhead to request paths. It also needed to support rich targeting (user ID, team, plan type, geography, arbitrary metadata), percentage-based rollouts, and real-time flag updates without service restarts.

Goals

  • Sub-millisecond p99 flag evaluation latency
  • Support for rich targeting rules and gradual rollouts
  • Real-time flag updates without restarts or re-deploys
  • A/B testing support with consistent assignment
  • Simple SDK for service teams to integrate
  • High availability — no single point of failure

Approach

The core design kept flag evaluation entirely in-process. Each service embedded a lightweight SDK that maintained a local, fully-evaluated copy of all flag rules in memory. This meant flag checks were pure in-memory lookups — no network hop, no database query.

Flag configurations were distributed via a push model backed by gRPC streaming. A central control plane streamed flag state changes to all connected SDK instances in real time, keeping local caches fresh with minimal lag. Redis served as the durable backing store and fan-out layer.

Consistent hashing was used to ensure stable assignment for percentage rollouts — the same user always ended up in the same bucket, making A/B test results reproducible and preventing flickering.

Key Challenges

Cold Start and Cache Warm-Up

Problem: On service startup, the in-memory cache was empty. Flag evaluations before the cache was populated could return incorrect defaults, potentially blocking user traffic.

Solution: The SDK blocked on initial flag load during startup, with a configurable timeout and a fallback to a locally persisted snapshot. This ensured services never started serving traffic with a cold cache.

Trade-off: Slightly slower cold starts, but correct behavior guaranteed from the first request.

Real-Time Propagation at Scale

Problem: Pushing flag updates to thousands of connected SDK instances simultaneously risked thundering-herd patterns on the control plane and backing store.

Solution: Used a tiered fan-out model with Redis pub/sub handling the initial broadcast and SDK instances reconnecting with jittered backoff on disruption. Control plane instances were stateless and horizontally scalable.

Consistent Assignment for A/B Testing

Problem: Percentage-based rollouts needed to be sticky — the same user should always see the same variant, even across different services and restarts.

Solution: Deterministic consistent hashing over a composite key (flag ID + user ID) ensured stable, reproducible bucket assignment with no external state required.

Results & Impact

  • Billions of flag evaluations per day
  • <1ms p99 evaluation latency
  • Real-time propagation of flag updates across the fleet
  • Adopted by all engineering teams as the standard rollout mechanism
  • Enabled safe, incremental delivery across dozens of major features

My Contributions

  • Led end-to-end architecture and design
  • Built the control plane and flag evaluation engine
  • Designed the SDK and client integration model
  • Implemented consistent hashing for A/B assignment
  • Built the real-time push infrastructure and fan-out layer
  • Drove adoption across engineering teams

Lessons Learned

Keeping evaluation in-process was the right call — it gave us latency guarantees no network-based system could match. The harder problem turned out to be operational: flag hygiene. As adoption grew, the number of stale and redundant flags accumulated quickly. Building tooling to surface and clean up unused flags became as important as the evaluation engine itself.
